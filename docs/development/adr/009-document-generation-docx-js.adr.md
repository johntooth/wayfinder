# ADR-009 â€” Document Generation: DOCX Template Upload + Placeholder Substitution

- **Status**: Accepted
- **Date**: 2026-05-19

## Context

Wayfinder generates compliance documents at the end of certain steps. The
constraints:

- Output must be **editable** by users downstream â†’ DOCX, not PDF.
- Generation must happen **server-side** â€” the AI key never reaches the
  browser.
- Flow admins must retain **full control over document formatting** â€” fonts,
  headers, footers, tables, and layout must match their organisation's
  templates exactly, not be re-generated by an AI.
- A document is a function of `(node.config.document_template_path,
  sessionMessages, flowContextDocs)` â€” pure inputs, no external state.
- Filenames must be predictable for filing in agency document stores.

## Decision

Flow admins upload a real `.docx` template file per document-generating node.
The template contains `{{placeholder_name}}` markers where the AI should fill
in values derived from the session conversation. The server substitutes each
marker, preserving all other formatting exactly.

Library: **`docxtemplater`** (MIT, core package) + **`pizzip`** for ZIP
handling.

### Node configuration

When a node has `output_type = 'generate_document'`, its `config` JSONB
stores:

```json
{
  "output_type": "generate_document",
  "document_template_path": "/tmp/node-templates/<nodeId>/<filename>.docx",
  "document_template_filename": "original-upload-name.docx"
}
```

The canvas config modal provides a `.docx` upload (max 5 MB). After upload,
the server extracts and returns the placeholder names found in the template
(e.g. `project_name`, `scope`, `budget`) as read-only chips so the flow admin
can verify the AI will be asked to fill in the right fields.

### Pipeline

1. Step completes (confidence â‰Ą 90, `readyToAdvance === true`, node
   `output_type = 'generate_document'`).
2. Server loads the template from `document_template_path`.
3. Extracts the set of `{{placeholder_name}}` keys from the template XML.
4. Calls `ILanguageModel.generateObject` with a Zod schema whose keys match
   the extracted placeholder names and whose values are `z.string()`. The
   prompt includes the full session transcript and flow context docs.
5. Passes the extracted keyâ†’value map to `docxtemplater` to produce a filled
   DOCX buffer, preserving all original formatting.
6. Writes the buffer to `/tmp/<sessionId>-<nodeId>-<isoDate>.docx`.
7. Calls a cheap model once to generate a 2-line summary for the document
   card.
8. Inserts a row into `app_documents`.
9. Chat UI re-renders to include the document card.

### Placeholder naming convention

The canonical convention (documented in the canvas UI tooltip):

- Use `snake_case` names: `{{project_name}}`, `{{evaluation_criteria}}`.
- Keep names short and descriptive â€” the AI receives them as JSON keys.
- A template with zero `{{}}` markers is valid; the document is returned
  unchanged (useful for cover-page templates that need no AI-filled content).

### Filename pattern

```
[FlowName]-[NodeName]-[SessionId8]-[YYYY-MM-DD].docx
```

`FlowName` and `NodeName` are kebab-cased. `SessionId8` is the first 8 chars
of the UUID.

### Download endpoint

`GET /api/documents/[documentId]` â€” requires authenticated session; verifies
the requesting user can read the parent session (own session, admin, or shared
viewer); streams the file from `storage_path` with `Content-Disposition`
matching `filename`. Returns `410 Gone` with `{ error, hint: "regenerate" }`
if the file is missing (server restart wiped `/tmp`).

### Why not store templates or generated docs in Postgres `bytea` or S3 at MVP

- Postgres `bytea`: works but adds row weight to common queries.
- S3 (or any object store): introduces an infrastructure dependency that the
  template does not currently have. Phase 4+ may add a generic
  `IObjectStorage` port and swap from `/tmp`.

For MVP, `/tmp` + documented limitation ("documents are lost on server
restart â€” regenerate by re-completing the step") is acceptable.

## Consequences

**Positive**

- Document formatting is 100% controlled by the flow admin â€” the AI only
  fills in text, never changes structure.
- `docxtemplater`'s core (MIT) is purpose-built for this use case; no custom
  parser to maintain.
- Pure-function pipeline (`(template, values) â†’ docx`) is easy to test with
  fixture templates.

**Negative**

- `/tmp` storage is ephemeral for both templates and generated files. Documented
  limitation; addressed in Phase 4+.
- If a `{{placeholder}}` in the template has no matching key in the AI's
  response, `docxtemplater` leaves it blank by default. Mitigated by showing
  extracted placeholder names to the flow admin in the canvas UI so they can
  verify coverage before publishing.
- The AI must return valid string values for each key; if it returns `null`
  for a required placeholder the document will have empty sections. Mitigated
  by prompting the AI to always provide a best-effort value.

## Tests

- Fixture test: a canned `.docx` template with three `{{}}` markers + canned
  placeholder values â†’ assert the produced DOCX contains the substituted text
  (extracted via `docxtemplater`'s own tag parser).
- Placeholder extraction test: verify `extractPlaceholders(templateBuffer)`
  returns the correct key set.
- Endpoint tests: own-session 200; another user's 403; missing file 410.
